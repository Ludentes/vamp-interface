# FFHQ rescues Solver C on squint

Addendum to `2026-04-28-solver-c-squint-feasibility-result.md`. Records
that swapping the corpus from Flux-rendered to FFHQ resolved the
Duchenne bundle the v1.1 diagnostic surfaced.

## Result

Script: `src/demographic_pc/solver_c_squint_feasibility_ffhq.py`.
Inputs: 26,040 FFHQ rows from `output/reverse_index/reverse_index.parquet`
(post arcface + non-empty FairFace demographics filter, from 70,000
total). Cell key: `(ff_race, ff_gender, ff_age_bin)`. η = 5
blendshape scalars (smile, gaze, blink, brow, cheek) + 16 ArcFace
PCs = 21 dims. Pair restriction: top-50 ArcFace cosine neighbors per
row within cell (uncurated FFHQ identities are unique by construction;
KNN approximates the "same identity" constraint with "similar-looking
same-demographic"). 89/125 cells used (skipped 36 with < 30 rows).
762,311 candidate pairs.

## Comparison vs Flux corpus

| | Flux v1.1 | FFHQ |
|---|---|---|
| corpus rows | 8,115 | 26,040 |
| candidate pairs | 4,870,376 | 762,311 |
| η dimensions | 31 | 21 |
| pairs above p90 | 487,038 | 76,231 |
| **eff. rank of selected Δη** | **3 / 31** | **16 / 21** |
| **PC0 variance share** | **59%** | **16%** |
| median \|Δθ\| selected | 0.832 (≈2.2σ) | 1.166 (≈3.0σ) |
| median √(Δη^T W Δη) selected vs all | 2.05 vs 4.84 (2.4×) | 3.80 vs 5.49 (1.4×) |
| σ_θ² rank stability | 1.0 | 1.0 |
| **all 3 criteria pass** | **fail (diversity)** | **pass** |

The diagnostic difference is what PC0 looks like:

```
Flux  PC0 (59%):  smile 31% + blink 25% + arc_pc4 23% + cheek 9% + gaze 9%
FFHQ  PC0 (16%):  blink 61% + smile 30% + small mixes
```

On Flux, PC0 captures most of the selected Δη and explicitly bundles
smile + cheek + identity-shape with blink — the Duchenne signature.
On FFHQ, PC0 captures a sixth of the variance, is dominated by blink
(eye closure independent of mouth), and the Δη budget spreads across
16 effective dimensions instead of collapsing onto 3.

## Why this works

FFHQ is a photographic, uncurated portrait dataset. Squint occurs
in real photos for many reasons that text-prompted Flux generation
does not produce: sun glare, sleepy expressions, focused
concentration, wincing, tight facial structure unrelated to mouth
state. The probability of finding "squint without smile" pairs is
high in the data distribution because the data distribution was not
generated by a text-encoder bundle.

The Flux failure was *not* a metric problem and *not* an η-coverage
problem. It was a corpus-distribution problem. v1.1's eigenvector
decomposition showed the bundle structure precisely; swapping the
corpus to one without that bundle structure resolved it.

## Caveat: residual smile loading

FFHQ PC0 still has 30% loading on smile (alongside 61% blink).
Total smile contamination of selected Δη ≈ 30% × 15.7% ≈ 4.7% of the
Δη variance budget. This is much smaller than Flux's
~31% × 59% ≈ 18%, but not zero. Two mitigation paths if a v9-squint
Path B trainer fed FFHQ pairs ends up bundling Duchenne:

1. Re-weight Σ_η to upweight the smile dimension (penalize Δsmile
   harder before pair selection) — cheap.
2. Hard filter: require |Δsmile| < threshold during selection —
   cheap, may reduce pair count.

Defer until evidence of a problem in the trained slider's eval. The
4.7% residual is plausibly within tolerance for a slider that doesn't
need to be perfectly disentangled.

## Other notes

- ArcFace PCA k=16 retains only 12% of variance — FFHQ identity is
  genuinely high-dimensional (every image is a different person).
  Identity-orthogonality is enforced primarily by the within-cell
  ArcFace KNN top-50 restriction (similar-looking same-demographic
  faces), with the 16 PCs catching residual.
- 26k FFHQ rows after filtering. The 44k drop is ArcFace detection
  failures (det_thresh=0.5 drops ~60% of FFHQ per the
  [arcface detection threshold doc](2026-04-27-arcface-detection-threshold.md))
  plus the empty FairFace cells (120 rows with no demographic
  prediction). Could recover some by lowering det_thresh, but 26k is
  ample.
- Median √(Δη^T W Δη) compression on FFHQ is only 1.4×, much less
  than Flux's 2.4×. The reason is encouraging, not concerning: η is
  genuinely less correlated on FFHQ, so there is less to compress.
  Selection still picks pairs with much larger \|Δθ\| (3σ vs 2.2σ).

## What this unlocks

The original three-route fork from the productive-failure doc
collapses:

1. **Composition route (Solver A)** — still the cheapest path; if it
   ships clean squint via `+squint − smile − cheek` at render time,
   we may not need any trainer. Test in parallel with this finding.
2. **Corpus rebalance** — *unnecessary*. FFHQ already has the
   unbundled distribution; rendering more squint-explicit Flux
   prompts would be redundant.
3. **Path C (latent φ surrogate)** — *demoted from "forced by squint"
   to "possible future thread."* The original argument for Path C was
   that squint's pair distribution was bundled and only per-input
   gradient supervision could escape. FFHQ's pair distribution is
   not bundled. Path B with FFHQ pairs is now the default trainer
   route; Path C remains a candidate for axes where even FFHQ is
   bundled.

## Proposed v9-squint Path B trainer port

Inputs:
- Top-1000 selected FFHQ pairs from
  `output/solver_c_ffhq/ffhq_squint_pairs.parquet` (selected=True).
- Each pair → (I_pos high-squint, I_neg low-squint), VAE-encode at
  train time.
- Trainer: port `extensions_built_in/image_reference_slider_trainer/`
  from ai-toolkit to Flux flow-matching. ~30–50 lines per the
  measurement-grounded slider plan
  (`2026-04-26-measurement-grounded-slider-plan.md` Path B,
  trainer_changes section).
- Hyperparameters: copy v8-glasses (r=32, α=1, η=4, lr=1.25e-4
  constant, ~2000 steps). Eye-mask peak need not be hot since the
  supervision signal is per-image, not text-derived; default
  peak=2.5.
- Eval: same 216-cell battery as glasses, with intended-axis metric
  swapped to `bs_eyeSquintLeft + bs_eyeSquintRight` from MediaPipe
  on rendered samples.

Risks specific to FFHQ pairs:
- The trainer learns squint in the FFHQ photographic distribution.
  Inference at the same Flux Krea base must reproduce the squint
  effect — the supervision needs to transfer to text-prompted
  generations. Mitigation: at eval, render the standard 9-prompt
  demographic battery and verify squint engages monotonically with
  multiplier.
- Cross-identity pairs introduce noise unrelated to squint. The
  trainer's `(z_neg − z_pos) / (2w)` averages out un-correlated
  identity differences across pairs; load-bearing assumption is that
  identity is uncorrelated with squint within the selected set. The
  ArcFace KNN restriction was designed to enforce that locally.
- MediaPipe blendshape noise on real photos is higher than on Flux
  renders. Some "high-squint" FFHQ pairs may be detector noise. The
  ArcFace top-50 KNN constraint partly mitigates by selecting
  similar faces (similar detector output), but a sanity pass through
  the top-J pairs is warranted before training.

## Cost so far

- v1.1 diagnostic on Flux: < 1 min, no GPU.
- FFHQ diagnostic: ~30 s, no GPU.
- One adversarial review pass + one corpus swap.
- Avoided: training a v9-squint Path B run on Flux pairs that would
  have produced a Duchenne LoRA. Avoided: rendering a corpus
  rebalance pass (~4–8 GPU-hours) that turns out to be unnecessary.
